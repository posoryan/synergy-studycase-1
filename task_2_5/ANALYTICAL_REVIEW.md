# Кейс-задача 2.5 — Аналитический обзор веб-приложения «Туристическое агентство»

**Объект анализа:** кейс-задача 2.4 — веб-приложение туристического агентства на стеке Delphi 10.2 + IIS + MS SQL Server.

**Дата:** 13.06.2026

## 1. Краткое описание системы

Разрабатываемая информационная система предназначена для автоматизации работы туристического агентства:

- **Публичная часть** — каталог опубликованных туров, форма онлайн-заявки на бронирование.
- **Административная часть** — авторизация менеджеров, управление турами, обработка заявок.
- **База данных** — MS SQL Server (`TourismWebApp`): справочники (Roles, Destinations, TourCategories, Users) и переменные данные (Tours, Bookings).

Архитектура: браузер → IIS → Delphi (IntraWeb / WebBroker) → FireDAC → MS SQL Server.

## 2. Анализ по критериям качества

### 2.1. Функциональность

**Оценка: удовлетворительная (с потенциалом улучшения)**

| Аспект | Состояние | Комментарий |
|--------|-----------|-------------|
| Просмотр каталога туров | Реализовано | Запрос `sp_GetPublishedTours`, фильтрация по `IsPublished = 1` |
| Бронирование | Реализовано | Модуль `BookingLogic.pas`, проверка свободных мест |
| Авторизация менеджеров | Реализовано частично | Есть модуль `AuthLogic.pas`, но пароль хранится как заглушка |
| CRUD туров | Описано в ТЗ | Требует реализации формы AdminForm |
| Валидация ввода | Частичная | Проверяются ФИО и количество человек; email/телефон не валидируются по формату |
| Обработка ошибок БД | Минимальная | Нет пользовательских сообщений при недоступности SQL Server |

**Выявленные проблемы:**
1. Нет автоматического обновления `SeatsBooked` при подтверждении бронирования.
2. Отсутствует email-уведомление клиенту о статусе заявки.
3. Нет пагинации каталога — при большом числе туров страница станет неудобной.

### 2.2. Производительность

**Оценка: хорошая для малых и средних нагрузок**

| Аспект | Состояние |
|--------|-----------|
| Индексы на FK и часто фильтруемых полях | Есть (`IX_Tours_StartDate`, `IX_Bookings_Status` и др.) |
| Хранимая процедура для каталога | `sp_GetPublishedTours` — снижает дублирование SQL |
| Кэширование | Отсутствует |
| Пул соединений | Не настроен явно в образце кода |

**Выявленные проблемы:**
1. При каждом HTTP-запросе создаётся новое подключение FireDAC (см. `WebModule_u.pas`) — на высокой нагрузке это узкое место.
2. Нет кэширования статического каталога туров (данные меняются редко).
3. HTML генерируется строками в коде — при росте функционала это замедлит разработку, но не критично для учебного проекта.

### 2.3. Удобство использования (юзабилити)

**Оценка: базовая**

**Сильные стороны:**
- Простая навигация: каталог → бронирование → подтверждение.
- Макеты экранов описаны в README задачи 2.4.
- Понятные сообщения об ошибках в `BookingLogic` («Недостаточно свободных мест»).

**Выявленные проблемы:**
1. Нет адаптивной вёрстки — на мобильных устройствах таблица туров может отображаться некорректно.
2. Нет индикатора загрузки при отправке формы.
3. Сообщения об ошибках авторизации не различают «неверный логин» и «неверный пароль» (это правильно с точки зрения безопасности, но нужно единое нейтральное сообщение).
4. Отсутствует хлебная навигация (breadcrumbs).

### 2.4. Безопасность

**Оценка: недостаточная для production, приемлемая для учебного проекта**

| Угроза | Текущее состояние | Риск |
|--------|-------------------|------|
| Хранение паролей | `HASH_PLACEHOLDER` — заглушка | **Высокий** |
| SQL-инъекции | FireDAC с параметрами (`:TourId`) | **Низкий** |
| XSS | HTML формируется из данных БД без экранирования | **Средний** |
| CSRF | Нет токенов в формах | **Средний** |
| HTTPS | Не настроен в учебном прототипе | **Средний** |
| Разграничение ролей | Таблица Roles есть, проверка на каждом action — не реализована | **Средний** |

**Выявленные проблемы:**
1. Пароли не хэшируются (bcrypt / PBKDF2).
2. Нет HTTPS — данные клиентов передаются открытым текстом.
3. Нет защиты от CSRF на форме бронирования.
4. Нет аудита действий менеджера (кто подтвердил/отменил заявку — поле `ProcessedBy` есть, но не везде заполняется).

### 2.5. Масштабируемость

**Оценка: ограниченная**

**Сильные стороны:**
- Разделение на справочники и транзакционные таблицы упрощает добавление новых направлений и категорий.
- IIS поддерживает несколько worker processes (web garden).

**Выявленные проблемы:**
1. Монолитная архитектура Delphi — сложно масштабировать горизонтально без load balancer и нескольких серверов.
2. MS SQL Server Express имеет лимит 10 ГБ на базу — для учебного проекта достаточно, для production нужна другая редакция.
3. Нет очереди сообщений для асинхронных задач (email, отчёты).

## 4. Предложения по устранению выявленных проблем

### 4.1. Функциональность

**Проблема:** не уменьшается количество свободных мест при подтверждении бронирования.

**Решение:** добавить транзакцию в хранимую процедуру:

```sql
CREATE PROCEDURE dbo.sp_ConfirmBooking
    @BookingId INT,
    @UserId INT
AS
BEGIN
    SET NOCOUNT ON;
    BEGIN TRANSACTION;

    DECLARE @TourId INT, @Persons INT;
    SELECT @TourId = TourId, @Persons = PersonsCount
    FROM dbo.Bookings WHERE BookingId = @BookingId AND Status = N'new';

    IF @TourId IS NULL
    BEGIN
        ROLLBACK;
        RAISERROR(N'Заявка не найдена или уже обработана', 16, 1);
        RETURN;
    END;

    UPDATE dbo.Tours
    SET SeatsBooked = SeatsBooked + @Persons
    WHERE TourId = @TourId
      AND (SeatsTotal - SeatsBooked) >= @Persons;

    IF @@ROWCOUNT = 0
    BEGIN
        ROLLBACK;
        RAISERROR(N'Недостаточно мест', 16, 1);
        RETURN;
    END;

    UPDATE dbo.Bookings
    SET Status = N'confirmed', ProcessedBy = @UserId
    WHERE BookingId = @BookingId;

    COMMIT;
END;
```

### 4.2. Безопасность — хэширование паролей

**Проблема:** пароль хранится как `HASH_PLACEHOLDER`.

**Решение:**
1. Использовать bcrypt или PBKDF2 при регистрации/смене пароля.
2. В `AuthLogic.pas` сравнивать `HashPassword(InputPassword)` с полем `PasswordHash`.
3. Добавить политику паролей: минимум 8 символов, цифра, спецсимвол.

### 4.3. Безопасность — HTTPS

**Проблема:** данные передаются по HTTP.

**Решение:**
1. Получить SSL-сертификат (Let's Encrypt или корпоративный).
2. В IIS: **Bindings → Add → https, port 443**.
3. Включить **Require SSL** для административных страниц.
4. Добавить redirect HTTP → HTTPS в `web.config`.

### 4.4. Безопасность — XSS и CSRF

**Проблема:** данные из БД выводятся без экранирования; формы без токенов.

**Решение:**
1. Экранировать HTML-сущности при выводе (`Server.HTMLEncode` / аналог в IntraWeb).
2. Генерировать CSRF-токен в сессии и проверять в POST-обработчиках форм бронирования и входа.
3. Установить заголовок `Content-Security-Policy`.

### 4.5. Производительность — пул соединений

**Проблема:** новое подключение FireDAC на каждый HTTP-запрос.

**Решение:**
1. Вынести `TFDConnection` в `TDataModule`, создаваемый один раз при старте ISAPI DLL.
2. Использовать `TFDManager` с пулом соединений FireDAC (`Pooled=True`).
3. Добавить кэш каталога туров на 5–10 минут (IntraWeb session cache или таблица с timestamp).

### 4.6. Юзабилити — мобильная версия

**Проблема:** таблица туров не адаптирована под смартфоны.

**Решение:**
1. Подключить CSS-фреймворк (Bootstrap 5) к HTML-шаблонам.
2. Заменить `<table>` на карточки туров с `@media` queries.
3. Добавить spinner при отправке формы (JavaScript `onsubmit`).

### 4.7. Масштабируемость

**Проблема:** монолитная архитектура.

**Решение (поэтапно):**
1. **Краткосрочно:** увеличить worker processes в IIS Application Pool; перейти на SQL Server Standard.
2. **Среднесрочно:** вынести API бронирования в отдельный REST-сервис (Delphi DataSnap или ASP.NET Core).
3. **Долгосрочно:** микросервисы (Catalog, Booking, Auth) + Redis для сессий + очередь (RabbitMQ) для email.

### 4.8. Сопровождаемость — разделение слоёв

**Проблема:** HTML генерируется в Pascal-коде.

**Решение:**
1. Использовать шаблоны IntraWeb (IWTemplateProcessorHTML) или внешние `.html` файлы.
2. Создать `TDataModule` для всех FDQuery — единая точка доступа к данным.
3. Ввести enum `TBookingStatus = (bsNew, bsConfirmed, bsCancelled, bsCompleted)`.

### 4.9. Тестирование

**Проблема:** нет автоматических тестов.

**Решение:**
1. DUnit-тесты для `BookingLogic.CreateBooking` с mock-соединением.
2. SQL-скрипт тестовых данных для граничных случаев.
3. Чек-лист ручного тестирования (см. README задачи 2.4):
   - бронирование при 0 мест → ошибка;
   - вход с неверным паролем → отказ;
   - XSS в поле ФИО → экранирование;
   - подтверждение заявки → `SeatsBooked` увеличился.

### 4.10. Переносимость

**Проблема:** привязка к Windows.

**Решение (если нужна переносимость):**
1. Переписать backend на ASP.NET Core или Python/FastAPI — API останется, фронтенд на HTML/JS.
2. Использовать Docker + Azure SQL / PostgreSQL.
3. Если Windows обязателен — документировать версии: Windows Server 2019+, IIS 10+, SQL Server 2019+, Delphi 10.2.

## 6. Заключение

Веб-приложение «Туристическое агентство» в текущем виде (учебный прототип) **выполняет базовые функции**: показывает каталог туров, принимает заявки, предусматривает авторизацию менеджеров. Архитектура понятна, база данных нормализована, SQL-скрипт и модули Delphi позволяют воспроизвести проект.

Основные риски для реального использования — **безопасность** (пароли, HTTPS, CSRF/XSS) и **целостность данных** (неатомарное обновление мест). Для production необходимо выполнить доработки из раздела 5, начиная с этапов 1–3.

Для учебной сдачи проект **соответствует требованиям задачи 2.4**: есть анализ рынка, SQL Server база, описание Delphi-приложения и данный аналитический обзор (задача 2.5).
